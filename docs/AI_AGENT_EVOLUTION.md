# 订单节拍器 → AI Agent 进化方案

## 一、当前系统 vs AI Agent 对比

### 当前系统：AI 辅助工具
```
用户操作 → 系统响应 → AI 给建议（文字） → 用户读完自己决定 → 用户手动执行
```

| 能力 | 当前状态 | 说明 |
|------|---------|------|
| 感知 | ❌ 被动 | 用户打开页面才触发 AI 分析 |
| 思考 | ⚠️ 浅层 | 只在用户请求时分析，无主动思考 |
| 决策 | ❌ 无 | AI 只给建议文字，不做决策 |
| 执行 | ❌ 无 | AI 不能操作任何数据，全靠人 |
| 学习 | ❌ 无 | 不会从历史结果改进决策 |
| 主动性 | ❌ 无 | 不会主动发现问题并通知 |

### 目标：AI Agent（自主代理）
```
Agent 持续监控 → 发现异常 → 分析根因 → 制定方案 → [人确认] → 自动执行 → 记录结果 → 下次更聪明
```

| 能力 | 目标状态 | 说明 |
|------|---------|------|
| 感知 | ✅ 主动 | 定时扫描所有订单，主动发现风险 |
| 思考 | ✅ 深度 | 结合历史、客户记忆、工厂画像做推理 |
| 决策 | ✅ 分级 | 低风险自动执行，高风险人确认 |
| 执行 | ✅ 有限 | 可发通知、分配任务、申请延期、标记状态 |
| 学习 | ✅ 渐进 | 记录每次决策结果，优化下次建议 |
| 主动性 | ✅ 全天候 | 每小时巡检，紧急事件即时通知 |

## 二、三阶段进化路线

### Phase 1：可执行建议（2周）
**核心改变**：AI 建议从"文字"变成"可点击的按钮"

当前：
```
AI: "建议将验货节点分配给秦增富"
用户：自己去找到那个节点，手动分配
```

Phase 1 后：
```
AI: "建议将验货节点分配给秦增富" [一键分配] [忽略]
用户：点击按钮，系统自动执行
```

**可执行的动作（白名单）**：
1. 分配里程碑负责人
2. 发送催办通知
3. 创建延期申请草稿
4. 标记节点为阻塞
5. 发送邮件提醒

**不可执行的动作（黑名单）**：
- 修改订单金额/价格
- 删除任何数据
- 直接联系客户
- 修改工厂信息
- 审批任何申请

**技术实现**：
- 新增 `agent_actions` 表记录 AI 提议 + 用户决策
- War Room 建议卡片增加执行按钮
- 每个执行动作都记录到 `milestone_logs`

---

### Phase 2：条件自动执行（1个月后）
**核心改变**：满足特定条件时，AI 可以自动执行低风险操作

**自动执行条件**：
| 条件 | 自动动作 | 理由 |
|------|---------|------|
| 节点超期 >3天 + 无人认领 | 通知管理员 + 角色负责人 | 低风险，纯通知 |
| ETD <48h + 未订舱 | 紧急通知业务+物流 | 时间紧迫 |
| 尾查 PASS + 未放行 >1天 | 自动启动放行节点 | 顺序推进 |
| 日报连续2天未提交 | 提醒跟单 | 保持数据新鲜 |

**仍需人确认**：
- 涉及客户沟通
- 修改交期
- 财务相关操作
- 跨部门协调

**技术实现**：
- 新增 `agent_executions` 表记录自动执行日志
- Cron 每小时运行 Agent 巡检
- 可配置的自动执行规则（管理员可开关）

---

### Phase 3：全能 Agent（3个月后）
**核心改变**：Agent 拥有工具集，可以自主推理+执行复杂任务

**Agent 工具集**：
```typescript
// 感知工具（只读）
get_order_status(orderId)        // 获取订单完整状态
get_customer_history(customer)   // 获取客户历史
get_factory_capacity(factory)    // 获取工厂产能
get_overdue_milestones()         // 获取所有超期节点
get_production_progress(orderId) // 获取生产进度

// 执行工具（写入，需授权）
assign_milestone(milestoneId, userId)     // 分配负责人
send_notification(userId, message)        // 发送通知
create_delay_request(orderId, reason)     // 创建延期申请
update_milestone_note(milestoneId, note)  // 更新备注
escalate_to_ceo(orderId, reason)          // CEO 升级
add_customer_memory(customer, insight)    // 添加客户记忆
```

**Agent 推理示例**：
```
Agent 巡检发现：
→ 订单 QM-037，产前样客户确认已超期 5 天
→ 查询客户记忆：该客户历史 3 次平均确认时间 8 天
→ 查询后续节点：生产启动在 3 天后到期
→ 推理：如果客户再 3 天不确认，生产启动必然超期
→ 决策：
  1. 自动执行：通知业务跟进客户（低风险）
  2. 建议执行：申请延期 5 天（需人确认）
  3. 记录：客户记忆新增 "产前样确认通常需 8-10 天"
```

## 三、安全机制设计

### 1. 权限分级
| 级别 | Agent 可做 | 需人确认 |
|------|-----------|---------|
| L0 | 读取数据、分析 | 无需 |
| L1 | 发送内部通知 | 无需 |
| L2 | 分配任务、更新备注 | 可选 |
| L3 | 创建延期申请、标记状态 | 必须 |
| L4 | 修改排期、联系客户 | 必须+管理员 |
| L5 | 修改金额、删除数据 | 禁止 |

### 2. 审计追踪
- 每个 Agent 动作记录：时间、推理过程、执行结果
- 独立的 `agent_audit_log` 表
- 管理员可回溯任意决策

### 3. 回滚机制
- 所有 Agent 执行的操作 48 小时内可回滚
- 回滚按钮在审计日志中

### 4. 熔断机制
- 单小时 Agent 执行超过 20 次 → 暂停并通知管理员
- 单订单 Agent 操作超过 5 次/天 → 暂停
- 任何报错 → 该订单 Agent 暂停 24 小时

## 四、实施优先级

| 步骤 | 改动 | 风险 | 价值 |
|------|------|------|------|
| 1 | War Room 建议加执行按钮 | 低 | 高 |
| 2 | 超期通知自动发送 | 低 | 高 |
| 3 | 日报提醒自动发送 | 低 | 中 |
| 4 | 节点自动推进（顺序完成时） | 中 | 高 |
| 5 | Agent 巡检 cron（每小时） | 中 | 高 |
| 6 | 客户记忆自动积累 | 低 | 中 |
| 7 | 全工具集 Agent | 高 | 极高 |
