# QIMO OS V1.1 — Product Philosophy（产品哲学）

> **Date**: 2026-06-30 · 视角：CPO + ERP 产品架构师 + AI-OS 产品负责人 + 外贸服装 COO。
> **这不是功能设计**，是**系统设计思想**。不碰代码/DB/架构/事件/微服务。
> **检验标准**：所有结论必须服务**一家真实外贸服装公司的每天运营**。保持真实、克制。

---

## 任务一 — 唯一核心业务对象：**Order**

不是"五大中心"，不是"模块"。如果只能有一个核心对象，它是 **Order（订单）**。

**为什么是 Order，不是别的：**
- **不是 Customer**：客户是"谁"。一个不下单的客户，不产生任何运营动作——没有料、没有生产、没有成本。客户是**资产的根**，不是**运营的核**。
- **不是 Quote**：报价是"提案"，不是"承诺"。报价可以有 100 个，公司不会为一个报价去采购、生产。
- **不是 Material**：物料是为订单而存在的手段。没有订单，物料无目的。
- **是 Order**：订单是公司对客户的**唯一承诺**——承诺"这个款、这个色、这个量、这个价、这个期"。**公司每天所有动作（采购/生产/质检/出运/收款/利润）都是为了兑现某一个订单的承诺。** 在真实系统里，这一点是物理事实：每个下游对象都挂 `order_id`。

> **但一个克制的 CPO 必须补一句**：Order 是**运营之核（spine）**；**Customer 是源（root，公司的资产）**，**Profit 是果（fruit，公司存在的目的）**。三者是一棵树：**客户为根 → 订单为脊 → 利润为果**。过度只盯 Order，会忘了"为什么要这个订单"（利润）和"订单从哪来"（客户关系）。所以核心对象是 Order，但 OS 必须把 Customer 与 Profit 也当**一等公民**。

## 任务二 — Order Center = Enterprise Operation Hub（你的判断：正确）

**这个观点正确。** Order Center 不是五中心之一，它是**企业运营中枢**。理由（哲学层）：

**为什么 Customer/Quote/PO/采购/生产/出运/财务/利润都必须围绕 Order 流动：**
因为它们**本质上都是"某个订单的一个侧面"**——
- 采购 = 某订单的"料的侧面"
- 生产 = 某订单的"做的侧面"
- 财务 = 某订单的"钱的侧面"
- 出运 = 某订单的"交付的侧面"
没有订单，这些"侧面"无所附着。Order 是**唯一把它们绑成一体的对象**。

| 数据归属 | 内容 |
|---|---|
| **只能 Order 拥有**（承诺真相） | 款色码 · 数量 · 交期 · 成交价 · 贸易/付款条款 · 订单生命周期 |
| **只能引用**（跨订单复用，不归 Order 拥有） | 客户 · 产品款 · 物料主数据 · 供应商——这些被多个订单共享，订单**引用其 id**，不复制 |
| **绝不能复制**（下游只绑 order_id） | 数量/单价/款色码——采购、生产、财务**绝不重存**一份，只挂 order_id 取真相 |

> 这就是 Constitution 01/02 的哲学落地：**一切挂 order_id；允许生命周期，禁止复制维护。**

## 任务三 — 五大中心 = 五个"看订单的镜头"（运营能力中心，不是系统）

五中心不是五个系统，是从**五个角度看同一条订单链**的运营能力。

| 中心 | 为什么存在 | 负责的价值 | 拥有 | 不能拥有 | 输出 | 服务谁 |
|---|---|---|---|---|---|---|
| **业务开发** | 把陌生客户变成可执行承诺 | 成交（Quote→PO） | 客户/报价 | 订单/采购/成本 | Approved Quote → Hub | 全公司的"进单口" |
| **订单执行(Hub)** | 把承诺准确·准时·盈利地兑现 | 兑现承诺 | Order/明细/生命周期/生产执行单 | 客户主数据/供应商/工艺/价审 | order_id+款色码+交期 → 各侧面 | 全部下游 |
| **采购** | 让料按时·按价·按量到厂 | 料的可得性 | 物料需求/采购/到料 | 客户/订单款色码/价审 | 料的状态 → 生产/财务 | 生产 |
| **生产** | 让大货按期·合格出厂 | 交付的确定性 | 工厂/排产/质检/出运 | 报价/客户要求/采购价 | 交付状态 → Hub/财务 | 客户(经Hub) |
| **财务** | 守住利润与现金 | 盈利与现金安全 | 价审/成本/利润 | 客户/订单/生产/采购执行 | 利润真相/告警 | 老板/全公司 |

## 任务四 — AI = Object Agent（你的判断：方向正确，但要克制实现）

**判断：Object Agent 比"部门 AI"更符合 AI Operating System。原因：**
- **部门会变，对象不变。** 部门重组、人员更替，但"订单""物料""客户"这些对象十年不变。把 AI 绑在**对象**上（而非部门），AI 就成了**对象的守护者**，服务所有触碰该对象的部门——这正是 Constitution"围绕业务对象、不围绕部门"的 AI 版。
- **一个对象，一个守护 Agent，全员共用**：Order Agent 看每一张订单的健康；采购、生产、财务都用它的判断，而不是各自养一个"部门 AI"。

**但作为克制的 CPO，必须纠三个偏，否则会过度设计：**
1. **Object Agent 是"思想模型"，不是"8 个 AI 微服务"。** 不要去建 8 个独立 Agent 产品（那是过度工程）。它是"每个对象有一个 AI 负责其健康/完整/风险"的**职责模型**，可共享底层、增量实现。
2. **真实系统里它已经发芽**：现有的**交付置信度引擎**本质就是 **Order Agent**（持续给每张订单算健康分+给"谁该做什么"）；quoter 的 RAG 本质是 **Quote Agent**。模型不是凭空发明，是把已有的东西命名清楚。
3. **Object Agent 仍守 Constitution 04**：它**监控、建议、预警、生成草稿**，但**绝不确认对象的真相转移**（Order Agent 永不替人确认订单）。

**Enterprise Agent Model（建议的对象 Agent 谱系，按价值优先）：**
| Agent | 守护对象 | 它持续做什么 | 现状 |
|---|---|---|---|
| **Order Agent** | Order | 算交付健康、报风险、提下一步 | ✅✅ 已在（置信度引擎） |
| **Quote Agent** | Quote | 估单耗/成本、生成报价、提毛利风险 | ✅ 已在（quoter RAG） |
| **Material Agent** | Material | 汇总需求、催料、缺口/尾货预警 | 🟡 部分（本轮汇总视图） |
| **Profit Agent** | Profit | 实时毛利、负毛利告警 | ✅ 部分（profit_snapshots/alerts） |
| **Supplier Agent** | Supplier | 历史价/比价/交付评分 | 🔴 缺（无供应商主数据） |
| **Customer Agent** | Customer | 跟进节奏、流失/投诉信号 | 🟡（customer_rhythm/matters） |
| **Delivery Agent** | Shipment | 出运准时/单据齐 | 🟡（里程碑级） |

> 这张表也回答了"AI 在哪是真执行、哪是只承诺"：**真执行的 Agent 都是已经在算某个对象健康分的**（Order/Quote/Profit）；缺的（Supplier）正是缺对象。

## 任务五 — 按 Business Object 重组数据链（真相/派生/AI建议/人工确认）

不按部门，按对象。每个对象标四性：

| 对象 | 性质 | 谁定真相 |
|---|---|---|
| Customer | **真相 SoT** | 人工确认 |
| Inquiry | AI 建议（草稿） | 人工确认后成真相 |
| Sample | 真相（轻） | 人工 |
| Quote | AI 建议→**真相(approved)** | **销售确认** |
| Customer PO | 证据（Evidence） | 客户给，人工核 |
| **PO Compare** | **派生**（diff，不存真相） | AI 产、人确认 |
| **Order** | **真相 SoT（核心）** | **业务确认** |
| 生产执行单 MO | 真相（翻译） | 跟单确认 |
| Material Requirement | **派生**（从 BOM 算，可重算） | 系统算 |
| Purchase | 真相 | **采购确认** |
| Material Arrival | 真相 | 仓/采确认 |
| Production/QC | 真相（节点） | 生产/质检确认 |
| Packing/Shipment | 真相 | 物流确认 |
| Payment | 真相 | 财务确认 |
| **Profit** | **派生**（forecast/live/final 投影，永不二存） | 系统算、财务看 |

> **哲学**：**真相对象**靠人工确认（有 Owner）；**派生对象**（PO Compare/MRP/Profit/各种汇总）**Derived-Never-Stored**；**AI 只在"建议→人工确认"那一步**出现，永不跨过确认闸门。

## 任务六 — 采购围绕 **Material**，不围绕 PO（你的判断：正确且重要）

**正确。** 采购每天做的——找料/找供应商/比价/合并/催料/收料/补料/尾货/退货/对账——**没有一件是"维护采购订单"**，**每一件都是"让某个物料流动到位"**。PO 只是物料一生中的**一个事件**。

所以**采购中心唯一核心对象 = Material**，它有自己的一生：
```
Material: 需求(Requirement) → 寻源(Source) → 下单(Purchase) → 在途 → 到料(Arrival) → 消耗 → 尾货(Leftover)
```
- **Purchase Order 降级为 Material 生命周期里的一个事件**，不是采购的中心。
- **Supplier 是引用对象**（Material 从某 Supplier 买来）——这也解释了为什么"没有 Supplier 主数据"是采购最大短板：Material 一生里"从谁买、买得好不好"无处落。
- 这与现有 **ADR-002（Material Requirement 为跨域脊柱）一致**——不是新发明，是把它升为采购的产品哲学核心。

> 采购中心的健康 = **每个物料的"到位确定性"**。Material Agent 的工作就是盯每个料：够不够、到没到、贵不贵、剩多少。

## 任务七 — 生产围绕 **Delivery**，不围绕 Factory/Task（你的判断：正确，且本就是项目的灵魂）

**正确——而且这恰恰是 QIMO 的原始灵魂。** 客户不关心工厂、不关心生产任务，只关心：**我的货，能不能按时、按量、按质收到。** 所以生产中心真正的对象是 **Delivery（交付）**。

```
Delivery 的各阶段：PMC/排产 → 产前样 → 大货 → 首查/中查/尾查(QC) → Packing → Shipment
```
PMC、QC、Inspection、Packing、Shipment **都只是 Delivery 的不同阶段**，不是各自独立的目的。

> 这不是新思想——CLAUDE.md 写的核心理念是"**卡风险，而不是走流程**"，Runtime Phase1 就叫"**Delivery Confidence（交付置信度）**"。**系统早就在按"订单能不能按时交付"思考，而不是按"流程节点完成没"。** 任务七只是把这个灵魂提到产品哲学的明面：**Production Center 的对象是 Delivery，KPI 是"准时·齐套·合格"，Delivery Agent = 现有置信度引擎。** 工厂/任务是手段，交付是目的。

## 任务八 — 财务升级为 **Profit Center**（你的判断：正确，但要划清边界）

**正确——在 QIMO OS 里，财务的对象是 Profit，不是记账。** 理由：
- **记账（AR/AP/凭证/总账）有它的家**——独立的财务系统。QIMO OS **不该重做账**（否则就是双真相源，违反单一真相）。
- **QIMO OS 里"财务"真正要管的是利润**：这单赚不赚、毛利够不够、负毛利要不要拦、成本有没有失控。这正是现有 `profit_snapshots`(forecast/live/final) + 毛利告警在做的事——**它本来就是 Profit-centric，不是 ledger-centric。**

> 所以：**在 QIMO OS 内，Finance Center → Profit Center**；其对象是 Profit，其使命是"守住利润与现金"，**记账留在财务系统**。这既升级了定位，又**避免 QIMO 重做账**（守单一真相）。利润才是企业最终输出，把这个中心命名为它真正守护的东西。

## 任务九 — KPI = Object Health（对象健康），由 Object Agent 持续监控

不输出部门 KPI，输出**对象健康**——每个对象问一句"它现在健不健康"，由它的 Agent 持续算：

| 对象 | 健康 = ? | Agent 怎么持续看 |
|---|---|---|
| Customer | 在成长还是在流失 | 复购/跟进逾期/投诉信号 |
| Quote | 转化得动、赚得到吗 | 成交率/响应时长/毛利质量 |
| **Order** | **能否准时·齐套·盈利交付** | **交付置信度分（已在算）** |
| Material | 会不会缺料/尾货堆积 | 到位率/缺口/尾货 |
| Supplier | 靠不靠谱 | 交付率/价格/质量 |
| Delivery | 会不会延期/不合格 | 准时率/一次合格率 |
| Profit | 这单/这月赚不赚 | 毛利/现金/回款 |

> **任务四 + 任务九 在这里合一**：**Object Agent 的本职，就是持续计算它所守护对象的健康 KPI 并预警。** "对象 + 它的健康分 + 它的 AI 守护者"是同一件事的三个面。**交付置信度引擎已经证明这套行得通**（Order Agent 每天给每张订单算健康分）——把同样的范式复制给 Material/Supplier/Profit 即可。

## 任务十 — QIMO OS V1.1 Product Philosophy（一句话 + 展开）

> **QIMO OS 是一个操作系统：每个业务对象有唯一真相、唯一 Owner、一个守护它健康的 AI Agent、和一个确认它关键转移的人——全部围绕 Order 这条脊柱流动，把"从客户到利润"的一条链，绑成公司每天的运营。**

五句话哲学：
1. **一条链**：从 Customer 到 Profit，公司只有一条价值链，订单是它的脊。
2. **一对象一真相**：每个对象一个 Owner、一份真相、一处录入；派生物永远 Derived-Never-Stored。
3. **AI 是对象的守护者**：Object Agent 持续算对象健康、报风险、出草稿——但**永不替人确认真相**。
4. **人确认关键转移**：报价定案、PO 转订单、采购下单、生产放行、付款——必经人。
5. **以交付与利润为终极**：所有动作最终回答两件事——**货能不能按时交到（Delivery）、这单赚不赚（Profit）。**

## 任务十一 — 相比 Five Centers Blueprint，升级了什么 / 不变什么 / 该改什么

| 维度 | Five Centers Blueprint（V1.0） | Product Philosophy（V1.1） | 评价 |
|---|---|---|---|
| 组织视角 | 按**部门/中心**组织（5 个工作台） | 按**对象**组织（中心是看对象的镜头） | ⬆️ **真升级**：从"部门协作"到"对象驱动" |
| AI 模型 | 部门 AI 助手（采购AI/财务AI…） | **Object Agent**（对象守护者） | ⬆️ 升级：更稳定、更 OS、更合宪 |
| 采购 | 围绕采购执行 | 围绕 **Material 一生** | ⬆️ 升级：PO 降为事件 |
| 生产 | 围绕里程碑/工厂 | 围绕 **Delivery** | ⬆️ 点明灵魂（本就如此） |
| 财务 | 成本/利润中心 | **Profit Center**（账留财务系统） | ⬆️ 升级 + 划清边界 |
| KPI | 部门 KPI | **Object Health KPI** | ⬆️ 升级：可被 Agent 持续监控 |
| 一条链/一真相/人工确认/Derived | 已有 | **不变**（只是上升到对象层表达） | ✅ 保持 |
| 完成度审计 / 每天用排序的缺口清单 | 已有 | **不变、依然有效**（这是现实） | ✅ 保持 |
| Order = Hub | 已提出 | **深化为"唯一核心对象"** | ✅ 深化 |

**该改的**：把三个中心**正名到它们真正守护的对象**——采购→Material、生产→Delivery、财务→Profit；把"部门 AI"重述为 **Object Agent**；把 KPI 重述为**对象健康**。这些是**命名与心智模型的改**，**不是推翻**，现有代码/表/中心**一行不用动**。

**该保持的**：Constitution 十条、一条链、一真相、人工确认闸门、Derived-Never-Stored、Order 为脊——全部不变。

## 任务十二 — COO 现实校验（克制：哲学服务于每天运营，不替代建设）

一个 20 年外贸 COO 的提醒：**哲学的价值，是让"下一步该建什么"变得显然，而不是让我们停下来欣赏模型。**

检验这套哲学是否有用，就看它**能不能解释为什么上一份审计的 P0 是对的**——能：
- **采购汇总接 UI** = 让 **Material Agent** 能被采购每天用（对象健康可见）。
- **quote→order 接线（origin_quote_id）** = 保护 **Order 对象的完整性**（一次录入，不重打）。
- **供应商主数据** = 给 **Supplier 对象**一个家（否则 Supplier Agent 无从谈起）。
- **Customer PO / PO Compare** = 给 **Order 对象一个受控入口**（Hub 从确认 PO 起，而非手工）。

> **每一个"每天会用"的缺口，都精确对应"某个对象的健康还看不见或入口还不受控"。** 这就是哲学落地的证据：V1.1 不增加一个新功能，但它让 V1.0 列出的 P0→P1 **每一件都有了"为哪个对象服务"的理由**。**先把这些对象补健康、补入口——QIMO OS 就从"能跑的订单系统"长成"外贸服装的 AI Operating System"。**
