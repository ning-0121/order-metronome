# 采购操作流 · 交互重设计清单(核料 → 归并 → 建PO → 收货)

> 触发:用户反馈"采购操作流不成熟——交互逻辑、页面跳转、归并"。
> 本文件是**基于实际代码**的交互审计 + 重设计,重点落地"归并双按钮"(用户拍板)。

---

## 一、当前流(实际代码)与不成熟点

### 流程骨架
```
/procurement(中心)
  → 待采购订单卡 → /procurement/verify/[orderId](核料 ProcurementItemsTab)
        → 「归并」(两步:dry-run 计划 → 勾选 create/refresh/cleanup → apply)
        → 「生成执行行」→ linesReady → 「去归采购单」
  → /procurement/po/new(建PO NewPurchaseOrderClient:平铺全部待归单,手动勾选)
  → /procurement/netting(跨订单 NettingClient:自动按键分组,每组一键建PO)
  → /procurement/po/[id](PO 详情:审批/下单/收货/催货)
```

### 🔴 根子问题:满屏原生浏览器弹窗(最伤体验)
| 位置 | 现状 |
|---|---|
| ProcurementItemsTab:259 补采购数量 | `window.prompt('补多少?只填数字')` —— 无校验、无单位选择、易误填 |
| ProcurementItemsTab:263 补采购原因 | `window.prompt('补采购原因')` —— 第二个连续 prompt |
| ProcurementItemsTab:275 驳回原因 | `window.prompt('驳回原因')` |
| ProcurementItemsTab:278/320 批准/批量确认 | `window.confirm(...)` |
| ProcurementItemsTab:250 移除图片 | `window.confirm(...)` |
| NettingClient:27/31/32 建单反馈 | `alert('请选择供应商')` / `alert('✅ 已建采购单')` |
| NewPurchaseOrderClient:43/44/50/59 | `alert` + `window.confirm`(合并提示藏在 submit 时的原生 confirm) |
- **影响**:原生弹窗无样式/无校验/无 loading/连续弹两三个,是"交互不成熟"的最大观感来源。
- **改法**:统一换成受控 Modal/表单(带字段校验、单位选择、loading、成功/失败 toast)。这一条 ROI 最高。

### 🟠 三个"归并/建PO"入口重叠,用户不知道用哪个
1. **核料内两步归并**(ProcurementItemsTab):把该订单的执行行归并成 procurement_items(款内),dry-run→勾选操作→apply。**勾的是"要不要执行 create/refresh/cleanup"三个操作**,不是勾物料——用户容易误解成"要手动挑物料"。
2. **建PO /po/new**:平铺**所有**待归单采购项,**手动逐行勾选**,选完选供应商建单。合并同料靠 submit 时 `window.confirm`。← **这是"必须勾选"的痛点来源**。
3. **跨订单 netting**:已**自动**按 consolidation_key 分组,每组一键建 PO。← 体验最好,但**每组只能建一张 PO**。

- **影响**:同一件事(把待采购的料变成采购单)有三条路、三种交互,新采购员懵。
- **改法**:**归并只留一个心智模型 + 双按钮**(见三)。

### 🟠 缺「按供应商归一张单」(你点名要的第二个按钮)
- 现状:netting 每组(=一种料)建一张 PO;po/new 勾一批建一张。**没有**"这几种不同料/不同色,都从辉念布行买 → 一张 PO 分行"。
- 大货实操最常要的就是这个。

### 🟡 页面跳转碎
- 核料 →「生成执行行」→ 亮「去归采购单」→ 跳 po/new(把该订单的行平铺)。多跳、且到了 po/new 又要重新勾。
- netting 与 po/new 各自独立,数据同源(待归单行)却两套 UI。

## 二、归并的正确心智模型(双按钮)

一句话:**系统自动分组,人只做两件事——「合并需求行」和「归到一张采购单」,分开两个按钮。**

### 按钮①「合并需求行」(line merge)= 凑量算总需求
- **规则(强制,护栏)**:只有**完全同键**(同物料 master / 同名+规格+类别 + **同颜色** + 同单位)才能并成一行。
- **交互**:系统**自动预分组 + 预勾**同键行;人点「合并需求行」即把这几行并为一条需求(数量求和)。
- **强制警示(你问的)**:若人手动多选了**不同键**的行去点合并 → **禁止 + 提示**「所选含 2 种颜色(黑/藏青)/2 种规格,不能并成一行。仅同料同色同规格可合并。」
- **主动提醒(补的)**:核料/队列页顶部飘一条「检测到 3 组可合并(面料A 跨2单、辅料B 跨3单),点此合并」——不用人去找。

### 按钮②「归到一张采购单」(PO grouping)= 同供应商一张单
- **规则(宽松)**:**同一供应商**即可,允许不同料/不同色**在一张 PO 里分行**(每行仍 peg 各自订单,核销口径不变)。
- **交互**:选一个供应商 → 把要给它的(可跨料跨色)行/组一起「归到一张采购单」→ 生成一张 PO 多行。
- 导出给供应商时,再叠加现有"合并同料"(consolidation_key)把同料并展示——**DB 行始终不合并,peg 不丢**(沿用现状)。

### 两个按钮的关系
- 先①(把碎行凑成干净的需求行),再②(把需求按供应商归成 PO)。
- ①对"量/需求"负责;②对"给谁下单"负责。互不干扰,概念清爽。

## 三、目标交互流(收敛成一条)

```
采购中心 / 核料页
  顶部主动提醒:「N 组可合并需求行」「M 项待归采购单」
  ↓
一个「待采购工作台」(替代 po/new 平铺 + netting 分开):
  - 待归采购项按 consolidation_key 自动分组展示(跨订单标"跨N单")
  - 每组:[合并需求行①](同键预勾,护栏拦异键)
  - 选供应商 → 勾若干组/行 → [归到一张采购单②](同供应商可跨料跨色)
  - 建单 → 受控反馈(toast + 跳 PO 详情)
  ↓
PO 详情:审批/下单/收货(收货入口就在本单,别再散落)/催货(补"催一次"按钮)/验收(合格/让步/拒收三选)
```

---

## 四、修复清单(分优先级)

**P0 · 体验止血(先做,ROI 最高)**
1. 把核料/netting/po-new 的所有 `window.prompt/confirm/alert` 换成受控 Modal/表单 + toast + loading + 字段校验(补采购数量要数字校验 + 单位选择;原因必填)。
2. 归并落地**双按钮**:①合并需求行(同键预勾+异键护栏警示)②归到一张采购单(同供应商跨料跨色)。

**P1 · 收敛与补全**
3. 三入口收敛成一个「待采购工作台」:自动分组 + 双按钮,替代 po/new 手动平铺;netting 的跨订单能力并进来。
4. 主动提醒:顶部飘"N 组可合并 / M 项待归单",不让人去找(DP-8)。
5. 补 PO 详情里的:收货入口(就在本单)、催一次按钮(留痕)、验收合格/让步/拒收三选(审计已指出这几处缺入口)。

**P2 · 打磨**
6. 供应商记忆:建单时默认带出该料上次的供应商/价(你已有"上次采购"记忆,接到选择器)。
7. 页面跳转减跳:核料「生成执行行」后直接进工作台并预选该订单的行,不用重勾。

## 五、要你拍板的
- 「待采购工作台」是**新开一个页**收敛三入口,还是就地强化 netting 页(它已是自动分组,最接近目标)?建议**强化 netting 页 → 升级成工作台**,po/new 退役,少造页。
- 双按钮的措辞:①「合并需求行」②「归到一张采购单」——用这个,还是你更顺口的叫法?
- 补采购/驳回等的受控表单,先做哪几个最高频的?
