## 1. submitBomToProcurement 升级后的完整时序

```
submitBomToProcurement(orderId):
  0. auth + 角色(sales/merchandiser/sales_manager/order_manager/admin)
  1. 读 order(quantity / factory_date / etd / order_no)— 缺 quantity → 拒绝
  2. 读 materials_bom(该订单 live 行)— 空 → 拒绝
  3. 读 order_cost_baseline.waste_pct(损耗,缺则默认 3)
  4. 读该订单 milestones(step_key, due_at)— 供 required_date 取锚点
  5. 顺序执行(无 DB 事务,逐步幂等 + 失败即返回不静默):
     a. 锁定 BOM:materials_bom.submit_status='submitted', submitted_at, submitted_by(全订单行)
     b. 冻结 Snapshot:
        - version =(该订单已有 max(version))+ 1
        - 若存在上一 approved 快照 → 标 superseded,新快照 supersedes_snapshot_id 指向它
        - insert material_package_snapshots(status='approved', submitted_by/at, approved_by/at=提交人, source_bom_count, snapshot_no=MPS-<order_no>-v<version>)
        - insert material_package_snapshot_lines(逐 BOM 行冻结复制 + loss_rate=waste_pct + bom_id 溯源)
     c. upsert material_plan(order_id 唯一):snapshot_id=新快照, plan_status='active', mrp_generated_at=now
     d. 重算 material_requirements(见 §5):每个 snapshot_line → computeMrp() → 写入
     e. 通知采购(Step A 已有,沿用)
  6. revalidate(/orders/[id])
  7. return { ok, plan_id, snapshot_version, requirement_count }
```
> 无原子事务(Supabase 客户端逐调用)。靠**幂等 + 顺序 + 失败显式返回**保证可重提修复(§10/§11)。

## 2. BOM 锁定规则
- 提交后:该订单全部 `materials_bom.submit_status='submitted'` = **锁定**。
- 锁定态:BomTab **禁用 新增/编辑/删除**(前端读 submit_status;B1 只定规则,UI 改属 B2,但锁定字段 B1 写)。
- 解锁唯一出口:**撤回修订**(§11)→ submit_status 回 'draft' + materials_bom.version+1。
- 权限:提交 / 撤回修订 = sales/merchandiser/admin。

---

## 3. Snapshot 生成规则
- **一次提交 = 一个不可变快照**;version 按订单自增(v1, v2…)。
- 头 `status='approved'`(v1 阶段:提交即批;独立审批步骤留作未来增强,字段已留 pending_approval)。
- `submitted_by/at` = 提交人/时刻;`approved_by/at` = 同提交人(v1);`source_bom_count` = 冻结行数;`snapshot_no` = `MPS-<order_no>-v<version>`。
- 行 `material_package_snapshot_lines` = 冻结当时每行:material_name/type/code/spec/color/placement/qty_per_piece/unit/**loss_rate(=order 的 waste_pct)**/suggested_supplier/sample_status/remarks/bom_id。
- 不可变:DB 层 `_lines` 无 UPDATE/DELETE 策略(B0 已设)。修订 = 出新版本,不改旧版本。

---

## 4. material_plan 创建 / 幂等规则
- **1:1 订单**(B0 `UNIQUE(order_id)`)。
- 首次提交 → insert;再次提交/修订 → update 同一行(snapshot_id 指向最新已批快照,mrp_generated_at 刷新)。
- 幂等键 = order_id。并发由 UNIQUE 防重。
- plan_status:`draft`(未提交)→`active`(已提交并算出需求)→`revising`(撤回修订中)→`active`(重提)→`closed`(订单结案,后续)。

---

## 5. material_requirements 生成规则
- **逐物料一行**:每个 snapshot_line → 一条 requirement(computeMrp 输出)。
- 重算策略(B1):重算前,**删除该 plan 下"无采购引用"的旧 requirements**,再按最新快照插入。
  - B1 中无 procurement_line_items 引用 requirement(红线),故全部可安全删除重建。
  - **⚠️ B3+ 规则(现在写明,防将来误删)**:`requirement_id` 已被 procurement_line_items 引用的需求 → **不删**,标记"快照已变更,请复核"(supersede 语义),只对未引用的重建。
- requirement 落:material_name/type/category/code/unit + MRP 数量字段 + 时间字段 + explain_json + status + version + snapshot_line_id + material_plan_id + order_id。
- status:`open`(待采购)/`fulfilled`(net=0 无需采购)/其余留后续。

## 6. Explainable MRP 纯函数规格
**纯函数,无 DB,无 Date.now(today 入参),可单测 —— 仿 `lib/runtime/deliveryConfidence.ts`。**

```
computeMaterialRequirement(input) -> output

input = {
  material: { name, type, category, code, unit, qty_per_piece, loss_rate, suggested_supplier },
  order:    { po_quantity },
  stageDate:        Date|null,     // 该 stage 锚点日(§7 取自 milestone)
  supplierLeadDays: number|null,   // §8
  inventoryQty: 0,                 // v1 固定 0
  reuseQty: 0,                     // v1 固定 0
  today: Date,
}

计算:
  gross_requirement = po_quantity × qty_per_piece
  loss_qty          = gross_requirement × (loss_rate/100)
  inventory_deduct  = -inventoryQty           // v1 = 0
  reuse_deduct      = -reuseQty               // v1 = 0
  net_raw           = gross_requirement + loss_qty - inventoryQty - reuseQty
  net_purchase_qty  = max(0, 向上取整(net_raw, 按 unit 精度))  // pcs→整数;kg/m→1位小数,宁多勿缺
  required_stage    = category 映射(§7)
  required_date     = stageDate
  order_by_date     = required_date 减 supplierLeadDays 个工作日(reuse lib/utils/date.subtractWorkingDays)
  timing_status     = §8
  explain_json      = §9

output = { gross_requirement, loss_qty, inventory_deduct, reuse_deduct, net_purchase_qty,
           unit, required_stage, required_date, supplier_lead_days, order_by_date,
           timing_status, explain_json }
```
- 缺 qty_per_piece → gross 无法算 → net=null,explain 标"缺单耗",**不阻断其他物料**。
- net=0(库存足够)→ status='fulfilled',explain 标"无需采购"。

## 7. required_stage → required_date 映射规则

**category → required_stage(默认表,可后续配置化):**
| category | required_stage | 理由 |
|---|---|---|
| fabric | cutting | 面料须在开裁前到 |
| print | cutting | 面料印花须在开裁前 |
| trim | sewing | 辅料影响车缝 |
| embroidery | sewing | 绣花影响车缝 |
| washing | packing | 水洗在缝后、包装前 |
| packing | packing | 包材影响包装 |
| service | other | 外发服务,时点另定 |
| other | other | 兜底 |

**required_stage → 锚点 milestone.due_at(reuse 现有里程碑日期 = One Data):**
| stage | 锚点(按优先级取第一个有值的)| 兜底 |
|---|---|---|
| cutting | production_kickoff | order.factory_date − 默认buffer |
| sewing | production_kickoff(+偏移,默认0)| 同上 |
| **packing** | **① packing 节点(packing_method_confirmed)→ ② shipment/etd − buffer → ③ factory_completion** | order.factory_date 兜底 |
| shipment | booking_done(无则 shipment_execute,无则 etd)| etd |
| sample | pre_production_sample_approved | factory_date − buffer |
| other | — | order.factory_date − 默认buffer |

> required_date = 对应里程碑的 due_at;缺则按优先级链下移;全缺 → required_date=null + timing_status='unknown'。
> **packing 特别说明**:包材真正影响的是**包装/出货**,不一定影响工厂完成日,所以**不直接用 factory_completion 兜太早**,优先 packing 节点 / 出货前 buffer。
> **print 特别说明**:print 默认按 `cutting` 处理(面料印花须在开裁前到);**但若未来是成衣后印花 / 特殊后整理印花,需人工改 stage** —— 此假设**必须写进 explain_json**(见 §9)。

## 8. supplier_lead_days / order_by_date / timing_status 计算规则
- **supplier_lead_days 默认值(按 CEO 拍板,工作日)**:
  | category | 默认交期(工作日)|
  |---|---|
  | fabric | 15 |
  | trim | 10 |
  | packing | 7 |
  | print | 10 |
  | embroidery | 10 |
  | washing | 7 |
  | service | 7 |
  | other | 10 |
- **order_by_date** = `subtractWorkingDays(required_date, supplier_lead_days)`(复用现有 date 工具,工作日口径,跳节假日)。required_date 为空 → order_by_date 为空。
- **timing_status**:
  - required_date/order_by_date 为空 → `unknown`
  - order_by_date < today → `late`(已晚)
  - today ≤ order_by_date ≤ today + 3 工作日 → `due_soon`
  - 否则 → `on_time`

## 8b. supplier_lead_days 来源追踪(采购可人工填写/修改)
**lead days 必须可人工覆盖,且记录来源与优先级。**

**优先级(高→低)**:`manual > quote > supplier_profile > default`

**字段设计(material_requirements 上)**:
| 字段 | 含义 |
|---|---|
| supplier_lead_days | 实际采用的交期(工作日)|
| lead_days_source | `default` / `manual` / `quote` / `supplier_profile` |
| lead_days_updated_by | 谁改的(manual 时)|
| lead_days_updated_at | 何时改的 |

**各阶段行为**:
- **B1**:系统用 category 默认 → `supplier_lead_days=默认值`,`lead_days_source='default'`。
- **B2/B3**:采购在 Material Requirement / 采购计划页**手填** → source='manual' → 系统重算 order_by_date / timing_status。
- **Step C**:供应商报价含 lead_days → 回写 → source='quote'(但 manual 优先级最高,不覆盖已 manual 的)。
- (将来 supplier_profile = 供应商档案默认交期,优先级在 quote 与 default 之间)

**explain_json 必须说明来源**,例如:
> "供应商交期使用采购手填 12 个工作日" / "供应商交期使用默认值 15 个工作日"

> ⚠️ **边界冲突(必须先解决)**:B0 建的 `material_requirements` **没有** `lead_days_source / lead_days_updated_by / lead_days_updated_at` 这 3 列;而 B1 边界写明"**不新增 migration**"。两者冲突。
> **解决方案(我的建议,见正文末"待你定")**:B1 实现阶段 lead 来源**只有 default**(manual/quote 是 B2/B3 的事),所以 B1 **把来源写进 explain_json(不加列、不加 migration,守住 B1 边界)**;等 B2 真正做"采购手填"时,再用 B2 的 migration 加这 3 列 + 回填 default。这样 B1 零 migration、设计完整、未来不返工。

```json
{
  "headline": "建议采购 主面料 2403kg,最晚 2026-07-02 前下单",
  "factors": [
    {"code":"gross","label":"PO 10000 × 单耗 0.265","value":2650,"unit":"kg"},
    {"code":"loss","label":"损耗 2%","value":53,"unit":"kg"},
    {"code":"inventory","label":"扣现有库存(v1=0)","value":0,"unit":"kg"},
    {"code":"reuse","label":"扣可复用余料(v1=0)","value":0,"unit":"kg"}
  ],
  "result": {
    "net_purchase_qty":2403,"unit":"kg",
    "required_stage":"cutting","required_date":"2026-07-12",
    "order_by_date":"2026-07-02","timing_status":"on_time"

  "next_action":"采购确认数量并询价",
  "assumptions":[
    "供应商交期使用默认值 15 个工作日(来源:default;采购可手填覆盖)",
    "印花默认按开裁(cutting)阶段处理;若为成衣后印花/特殊后整理,需人工改阶段"

  "lead_days_source":"default",
  "inputs":{"snapshot_version":1},"computed_at":"<iso>"

```
**explain_json 强制项**:① `assumptions` 必含**当前 lead_days 来源说明**(default/manual/quote)+ 数值;② 当 `category='print'` 时,必含上面那条"印花默认开裁、后印花需人工改阶段"的提示;③ `lead_days_source` 字段冗余进 explain(B1 恒为 default)。

## 10. 异常场景处理
| 场景 | 处理 |
|---|---|
| BOM 为空 | 拒绝提交,不建快照 |
| 订单缺 quantity | 拒绝(无法算 gross)|
| 物料缺单耗 qty_per_piece | 该行 net=null + explain 标"缺单耗",不阻断其它物料 |
| 缺阶段里程碑日期 | required_date 走兜底;再缺 → null + timing_status='unknown' |
| 缺供应商交期 | 用 category 默认 + explain 标"默认值" |
| 库存足够 net=0 | status='fulfilled' + explain 标"无需采购" |
| 部分步骤失败(快照建了、需求没算完)| 不静默;返回 error;**重提幂等补齐/重算**,不留半成品阻断 |
| 通知采购失败 | fire-and-forget,不阻断提交(Step A 同口径)|

## 11. 重复提交 / 撤回修订 / 重新提交规则
- **重复提交(BOM 未变再点)**:比较 live BOM 与最新快照;**一致 → 不建新版本,仅重算 requirements(刷新 MRP,因交期/日期可能变)**;不一致 → 视为修订。
- **撤回修订**:`materials_bom.submit_status='draft'`(解锁)+ `version+1`;快照/计划保留(历史);`material_plan.plan_status='revising'`。**不删任何快照/需求。**
- **重新提交(改后再提交)**:version 已 bump → 新快照 v(n+1) approved,上一 approved 标 superseded;plan.snapshot_id 更新;按 §5 重算 requirements(B1 全部可删重建;B3+ 保留有采购引用的)。
- 并发:plan 用 order_id unique 防重;requirement 用 version 乐观并发,冲突重试 1 次。

## 13. 验收标准
提交采购后:
1. `materials_bom.submit_status='submitted'`(锁定)。
2. `material_package_snapshots` 新增 1 条(version 正确)+ `_lines` = BOM 行数。
3. `material_plans` 该订单恰 1 条(snapshot_id 指向新版,plan_status='active')。

5. 采购收到通知。
6. 重提/修订/撤回 行为符合 §11(不产生垃圾版本、不误删)。
7. 采购中心 / Supply Chain Tab / ProcurementTab **无任何变化**;`procurement_line_items` **零新增**。
8. MRP 纯函数单测通过(`scripts/test-mrp.ts`:给定输入→预期 net/日期/explain)。
9. build + check 通过。
