## 0. 绑定约束 — AI 边界(用户 2026-06-28 拍板,呼应 Constitution 06)
**AI 永远不得直接生成 Manufacturing Order / 生产任务单,不得从 PO/Tech Pack/附件直接生成可执行生产单。**
- AI 最多只能:**OCR / 字段建议 / 遗漏检查 / 风险提醒 / 翻译 / 草稿提示**。
- **所有 AI 结果必须人工确认后**,才能进入 Manufacturing Order。
- 生产任务单**只能来自结构化数据**:Customer Order + Material Package + Manufacturing Order。
- O2 第一段**完全不接 AI**(连草稿都不做);AI 辅助留到后续阶段,且必经人工确认闸门。

## 1. 范围(O2 第一段,最小闭环)
1. 新建 1:1 卫星表 `manufacturing_orders`(migration)。
2. 订单详情页新增「🏭 生产任务单」Tab。
3. 结构化录入 MO 的**翻译字段**(印绣要求 / QC 重点 / 特殊要求 / 风险提醒)。
4. **生命周期状态**(draft → reviewing → confirmed → executing → closed)。
5. Tab 内**组合展示**已有真相(产品/数量/款色码/原辅料包/交期/包装/附件),**不复制**。
6. 一键**生成生产任务单文档**(从结构化 MO + 绑定真相,复用 exceljs)。

**第一段不做**:AI(连草稿)/ 多版本审批 / 工艺·SMV·IE·MES·吊挂(永不做,归生产域)/ 拆旧 legacy 生成器 / 模板引擎泛化 / 生产域回传。

## 3. 数据模型 — `manufacturing_orders`(1:1 卫星表,决策 A1)
**只持有"把客户需求翻译成工厂执行语言"的新字段 + 生命周期;其余全部绑定现有真相,零复制。**

| 列 | 类型 | 说明 |
|---|---|---|
| id | uuid PK | |
| order_id | uuid FK→orders **UNIQUE** | 1:1 绑定 Customer Order |
| mo_no | text | 生产任务单号(生成,如 `MO-{order_no}`)|
| status | text CHECK(draft/reviewing/confirmed/executing/closed) DEFAULT draft | 生命周期(Constitution 03)|
| print_embroidery_requirements | text | 印绣要求(新)|
| qc_focus | text | QC 重点(新)|
| special_requirements | text | 订单级特殊要求(新;区别于 materials_bom 的逐料 special_requirements)|
| risk_notes | text | 风险提醒(新)|
| factory_packing_instructions | text | 内部生产执行包装说明(新;**独立于 orders.packing_requirement** 客户原始要求)|
| factory_notes | text | 其他下厂说明(新,兜底)|
| confirmed_at / confirmed_by | timestamptz / uuid | **内容确认**留痕 |
| released_to_factory_at / released_to_factory_by | timestamptz / uuid | **正式下发工厂执行**留痕(对应 status→executing,与 confirmed 区分)|
| created_by / created_at / updated_at | | |

**绑定(不复制,展示/生成时 join):**
- 产品/款号/数量/交期 ← `orders`(product_name/style_no/quantity/etd/factory_date);**客户原始包装要求** ← `orders.packing_requirement`(MO 的 factory_packing_instructions 是内部翻译版,两者各自单一真相)
- 款×色×码 ← `order_line_items`
- 原辅料包 ← `materials_bom`(Material Package,O1)
- 附件 ← `attachments`/`order_attachments`

> 包装要求已在 `orders.packing_requirement`(单一真相)→ MO **绑定并可就地编辑 orders 的该字段**,不在 MO 复制一份。
> RLS:沿用 `auth.uid() IS NOT NULL`(SELECT/INSERT/UPDATE);幂等;纯加法;回滚 = DROP TABLE。

## 4. 生命周期(Constitution 03)
```
draft（业务起草）→ reviewing（跟单复核）→ confirmed（确认,留痕）→ executing（已下生产）→ closed（完成）
```
- 第一段:状态可前进/回退(简单按钮),不做强审批流。
- `confirmed` 记 confirmed_at/by(人工确认闸门,呼应"AI 结果须人工确认")。
- 不阻断现有里程碑/放行逻辑(独立轻量状态)。

## 5. UI — 订单详情页「🏭 生产任务单」Tab
- 顶部:MO 号 + 生命周期状态条(可推进)。
- **组合区(只读绑定)**:产品/款号/数量/交期 · 款色码表 · 原辅料包摘要(🔗 行数)· 包装要求(可编辑→写 orders)· 附件入口。
- **录入区(MO 新字段)**:印绣要求 / QC 重点 / 特殊要求 / 风险提醒(随手保存)。
- 底部:**「生成生产任务单」按钮** → 从结构化 MO + 绑定真相导出 Excel(复用 exceljs)。
- 空态:「+ 创建生产任务单」(建 manufacturing_orders 行)。

## 6. 生成器(决策 C)
