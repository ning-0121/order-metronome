# P1′ 设计 — Procurement Item(采购核料项)

> **Status**: 设计待审(重写版)。migration 草案 `DRAFT_20260628_p1_procurement_items.sql`,未执行。代码未写。
> **Date**: 2026-06-28 · 视角:ERP 首席架构师 + 服装供应链 + 制造业数字化。遵守 `../00-Constitution/Constitution.md`(01/02/03/04)。
> **定位**:Procurement Item(中文 **采购核料项**)是采购**每天真正工作的对象** —— 同订单内按 **物料身份 + 颜色 + 单位** 自动归并后的**采购确认项**。它不是 PO、不是供应商、不是到料。

---

## 0. 废弃旧方案(P1 → P1′)
旧 `procurement_reviews`(1:1 挂每条 material_requirement,锚 requirement_id)**已废弃**:
- 错误:采购不是逐 BOM 行确认,而是按**归并后的物料**工作。
- 旧 migration `20260628_p1_procurement_reviews.sql` **已删除**(从未执行、未跟踪,零损失)。
- 核料归并 与 采购确认 **合并为同一对象 = 采购核料项**(采购看的就是「一个物料 + 总需求 + 来源明细 + 最终采购方案」,本身即 Review)。

## 1. 审计结论(采购域现状 + 关键 schema)
| 对象 | 现状 | 职责 |
|---|---|---|
| `materials_bom` | O1 已建 | **Material Package(业务)**;`qty_per_piece` = **开发单耗** |
| `material_requirements`(B0/B1)| 已建 | **系统需求**(每 BOM 行);**有** name/material_type/category/material_code/unit/net_purchase_qty/required_date/order_by_date/status/version/snapshot_line_id;**无** color/specification/master_id |
| `material_package_snapshot_lines` | 已建 | 冻结物料行;**有** color/specification/qty_per_piece(开发单耗)/bom_id(→materials_bom.master_id)|
| `procurement_line_items` | legacy | **采购执行 PO + 到料对账** |
| 供应商主数据 | ❌ 无 | 仅文本 |

**关键事实**:归并要的 color / 开发单耗 / master_id 都不在 material_requirements 上 → 归并时 **join `material_package_snapshot_lines`(经 snapshot_line_id),再经 bom_id→materials_bom 取 master_id**。

## 2. P 域多层关系(全景)
```
Customer Order (业务)
 └ Material Package / materials_bom            【业务】开发单耗 qty_per_piece + 物料+色+特殊要求
   └ Material Requirement / material_requirements 【系统】每 BOM 行 MRP(开发单耗→net,时间分段,可重算投影)
     └ 🆕 Procurement Item / 采购核料项          【系统归并 + 采购确认 = P 域核心】
         · 系统:按 物料身份+颜色+单位 归并 N 条 requirement → 1 项,live 读来源明细
         · 采购:确认 大货单耗→自动算量 / 供应商 / 价 / 决策 / 状态
         └ 🆕 Purchase Order / 采购订单            【采购执行】下单(演进 procurement_line_items,P2)
           └ Receiving 到料 → Warehouse 库存 → Production 领料  【仓库/生产,Phase2】
```

## 3. 核心对象:采购核料项 `procurement_items`(1 项 / 归并后物料)
锚 **稳定物料身份**(非易失 requirement_id)。requirement 重算 → 采购项按身份**重新 peg**,采购确认不丢,只标 `needs_reconfirm`。

## 4. 归并规则(consolidation key)
**范围:同订单内**(跨订单核料 = P5)。归并键(app 计算 `consolidation_key`,`UNIQUE(order_id, consolidation_key)`):
```
key = order_id
    + material identity( material_master_id 优先;无则 lower(material_name)+'¦'+lower(specification)+'¦'+category )
    + lower(color)
    + lower(unit)
```
- material identity 优先 `material_master_id`;无 master(临时/未挂)用 `material_name + specification + category`。
- 同 color、同 unit 才归并(不同单位 kg/卷 不并)。
- 归并源 = `material_requirements ⋈ snapshot_lines`(取 color/spec/开发单耗)`⋈ materials_bom`(取 master_id)。
- 系统**自动归并**(采购不开 Excel 加总):一次 action 按订单算各 requirement 的 key → 分组 → upsert procurement_items;某 key 不再有 requirement → 标 `needs_reconfirm`。

## 5. 来源明细 pegging(req 5)
**一个采购项 → 多条 material_requirements(N:1),live 读取,不存死链**(避开 requirement 易失)。
- 来源明细 = `SELECT material_requirements ⋈ snapshot_lines WHERE order_id=X AND 派生key=item.consolidation_key`。
- UI 展开看到每来源:物料/颜色/部位 · **开发单耗**(snapshot.qty_per_piece)· 件数 · 需求量(net_purchase_qty)。
- `confirmed_source_snapshot jsonb`:确认时把来源明细**快照**进采购项(审计 + 判过期:live 来源 vs 快照不一致 → needs_reconfirm)。
### ⚠️ 5.1 P1′ 已知限制(必须明确记录,用户 2026-06-29 拍板选 A)
- **当前来源展开只能看到 `materials_bom` 行来源**(物料/颜色/部位/开发单耗/件数/需求量)。
- **暂时不能精确显示 Legging / Bra / Short 产品拆分** —— 因为 `materials_bom` 现为订单级,**未挂 `order_line_item_id`**(无产品维度)。
- **后续当 `materials_bom` 支持 `order_line_item_id` 后,再增强来源展开**(精确到产品)。这是 **O 域后续优化,不夹进 P1′、不改 O1**。
- **选 A 的理由**:订单系统在线使用;P1′ 核心价值是先解决"采购自动归并 + 大货单耗",不能为来源展开更漂亮而打断主线。

## 6. 大货单耗 + 数量模型(req 6,采购永不手算)
| 字段 | 来源 | 说明 |
|---|---|---|
| `development_consumption` | **业务/系统预填(只读)** | 开发单耗代表值(来自来源行,主导/加权)|
| `production_consumption` | **采购填** | 大货单耗(排料/缩率,通常 > 开发单耗)|
| `procurement_loss_pct` | 采购填 | 采购损耗% |
| `safety_stock_qty` | 采购填 | 安全库存 |
| `total_required_qty` | **系统**(归并)| Σ来源 net_purchase_qty(开发单耗算出的净需求,准确按各产品)|
| `suggested_purchase_qty` | **系统算** | = `净需求 × (production_consumption / development_consumption) × (1 + procurement_loss_pct) + safety_stock_qty`,再**按 MOQ / 采购单位取整** |
| `final_purchase_qty` | **采购确认** | 采购量**权威**,可**覆盖系统建议值**(取代 material_requirements.confirmed_qty)|

> 大货单耗通过「大货/开发 比」放大需求,既保留 total_required_qty 的逐产品准确性,又让采购的大货单耗驱动算量。per-source 大货单耗(更精)留后续。

## 7. 采购确认字段(req 7)
- **Supplier**:confirmed_supplier_name · backup_supplier_name · supplier_contact · lead_days · moq · purchase_unit
- **Price**:unit_price · currency(默认 CNY)· tax_rate · price_inclusive_tax · quote_date
- **Decision**:is_substitute · substitute_reason · is_split · is_outsourced · risk_flag · risk_note · procurement_notes

## 8. 字段所有权(Constitution 04,四层不混)
| 层 | 所有者 | 字段 | 禁止 |
|---|---|---|---|
| Material Package(materials_bom)| 业务 | 物料/色/位置/**开发单耗**/特殊要求 | 不碰采购 |
| Material Requirement | 系统 | net/required_date(开发单耗算)| 人不编辑 |
| **采购核料项 procurement_items** | **采购** | 大货单耗/损耗/安全库存/MOQ/供应商/价/决策/最终采购量/状态 | 不改业务需求、不下单 |
| Purchase Order(procurement_line_items)| 采购执行 | ordered/received/difference | — |
| Receiving/Warehouse | 仓库 | 到料/入库/库存 | — |

## 9. 生命周期(status)
```
draft(系统归并生成)→ reviewing(采购核对大货单耗/损耗/安全库存)
 → confirmed(采购确认 供应商/价/final_purchase_qty,记 confirmed_by/at + source_version)
 → ordered(生成/关联 PO,P2)→ partially_received → completed(到料,P3)→ closed
```

## 10. needs_reconfirm + 版本
- `needs_reconfirm boolean`:requirement 重算后该 key 的来源/总量变化、或物料消失 → true,UI 显「⚠ 需重新确认」。
- `source_snapshot_version` / `source_requirement_version`:确认时基于的 snapshots.version / requirements.version,判过期。

## 11. 修宪:不动,新增 ADR-004
**Constitution 保持冻结 10 条。** 新增 **ADR-004 — Procurement Domain 分层与核料原则**:
① 采购按「归并后的物料」工作,核料归并是系统职责(采购不手工加总);
② 开发单耗(业务)≠ 大货单耗(采购),采购量系统自动算;
③ 采购四层不混:Material Package(业务)/ 采购核料项(采购确认)/ Purchase Order(执行)/ Receiving(仓库);
④ 采购核料项锚物料身份,非易失需求 id。
> 验证稳定后,「四层不混」可再考虑升级进 Constitution —— 现在不动。(ADR 文件待 P1′ 定稿后单独建。)

## 12. 红线(P1′ 设计 + 编码都守)
